# Extension Subsystem

Compiled TS → JS lives in `extension/services/`, `extension/bin/`, `extension/hooks/`, `extension/types/`. Source in `extension/src/`. Tests in `extension/tests/` via `node --test`. Rebuild with `npx tsc` from `extension/`.

## Trap Doors

- `src/bin/council-publish.ts` — INVARIANT: every `execFileSync('gh', …)` passes `timeout`, threaded via `PublishOptions.ghTimeoutMs`. BREAKS: network hang stalls publish indefinitely. ENFORCE: hang-path tests using `__hang__` / `auth: 'hang'` / `hangOnCall` sentinels.
- `src/services/scope-resolver.ts` — INVARIANT: every `spawnSync('rg'|'grep', …)` in `findImporters` passes `timeout` (default `FIND_IMPORTERS_TIMEOUT_MS = 30_000`), threaded via `computeOneHop({ findImportersTimeoutMs })`; inline `diff:<ref>` scope bases must come from the flag itself, never from `scopeBase`/default-base fallback; per-phase `refreshScope()` must reapply the original `target` subtree clamp before persisting refreshed `allowed_paths`. BREAKS: wedged rg/grep stalls scope resolution silently; `diff:HEAD`/`diff:<release-branch>` silently reviews the wrong files and subsystems; later phase refreshes can widen scoped anatomy/szechuan review outside TARGET and misroute gate failures. ENFORCE: `scope-one-hop-hang-guard.test.js`, `scope-pipeline.test.js` diff:HEAD regression, and `scope-pipeline.test.js` target-clamp refresh regression.
- `src/bin/plumbus-frame-analyzer.ts` — INVARIANT: every `spawnSync('bun', …)` in `parseDotViaBun` passes `timeout` (default `BUN_TIMEOUT_MS = 30_000`). BREAKS: wedged bun stalls generative-audit pipeline silently. ENFORCE: `plumbus-frame-analyzer-hang-guard.test.js`.
- `src/services/pickle-utils.ts` — INVARIANT: macOS `osascript` shell-outs route through `displayMacNotification` (default `NOTIFICATION_TIMEOUT_MS = 5_000`); never call `spawnSync('osascript', …)` directly. BREAKS: wedged Notification Center blocks `process.exit`, leaks cancel markers, hangs tmux pane. ENFORCE: `notification-hang-guard.test.js` via `spawnSyncFn` seam.
- `src/bin/spawn-morty.ts` — INVARIANT: session wall-clock and backend routing must both read the same crash-recovered state snapshot via `StateManager.read()`; never clamp timeout from a raw `state.json` parse while backend routing uses `loadBackendFromSession()`. BREAKS: a recovered higher-iteration tmp can route the worker through one backend while the timeout budget still comes from stale base state, letting the worker overrun the live session wall clock. ENFORCE: orphan-tmp timeout/backend split-brain regression in `spawn-morty.test.js`.
- `src/bin/spawn-refinement-team.ts` — INVARIANT: refinement timeout selection, session wall-clock clamp, and codex-downgrade warning must all read the same crash-recovered snapshot via `StateManager.read()`; never size refinement workers from a raw `state.json` parse. BREAKS: a recovered higher-iteration tmp can hide the live timeout budget or backend preference, so refinement workers overrun the real session wall clock and suppress the warning that codex was requested but refinement is claude-only. ENFORCE: orphan-tmp timeout/backend recovery regression in `spawn-refinement-team.test.js`.
- `src/bin/microverse-runner.ts` — INVARIANT: auto-commit rescue stages tracked + untracked files via the shared staging helper, honoring `docs/`/`prds/` exclusions; per-iteration convergence gates must capture `gate/baseline.json` from the clean pre-iteration tree before the first worker commit, never from the first post-change diff; stale baselines must be recaptured from the clean loop-start tree before reuse, honoring `baseline_max_age_iterations` / `baseline_max_age_seconds`; they must baseline + replay `tests` alongside `typecheck`/`lint`; and every normal `state.json` control-flow read (startup, backend refresh, loop liveness, rate-limit wait) must go through `StateManager.read()`, never raw `JSON.parse(fs.readFileSync(...))`. BREAKS: `git add -u`-only drops new test files and first-time `CLAUDE.md`, then misclassifies as stall; first anatomy regression gets silently accepted as the baseline and the gate false-greens the iteration that introduced it; resumed sessions can keep subtracting obsolete failures forever if stale `gate/baseline.json` is reused; pure test regressions slip through green when the per-iteration gate omits `tests`; interrupted writes can strand newer `active:false` / `backend:'codex'` state in `state.json.tmp.<pid>`, so raw reads keep the loop running after cancellation or judge the baseline on the wrong backend. ENFORCE: untracked-file path in `microverse.test.js`, first-regression bootstrap + stale-baseline refresh coverage in `anatomy-park-baseline-gate.test.js`, and orphan-tmp state promotion coverage in `microverse.test.js`.
- `src/bin/council-publish.ts` (directive/stack parity) — INVARIANT: every non-trunk branch in `council-stack.json` has a matching `directive.branches` entry; missing entry → `outcome: 'failed'`, no body file, no `.published` marker. BREAKS: silent "empty findings" comments hide dropped fan-out shards. ENFORCE: `stack branch absent from directive` test in `council-publish.test.js`.
- `src/bin/pipeline-runner.ts` — INVARIANT: scoped anatomy/szechuan phases pass `scope.json` into `init-microverse` so `microverse.json.allowed_paths` survives setup; when `refreshScope()` returns `null` because the phase was already entered on resume, setup must reuse the persisted `scope.json` instead of widening back to unscoped review. BREAKS: repo-wide gate failures get treated as in-scope and scoped resume runs silently expand from the target organ to the whole body. ENFORCE: `anatomy-park-scope.test.js` and `szechuan-scope.test.js`.
- `src/bin/finalize-gate.ts` — INVARIANT: synthetic gate failures (`<timeout>`, `<lock-timeout>`, `<workingdir-drift>`) stay in-scope even when `allowed_paths` is present; only real repo paths go through path-based scope filtering; `working_dir` and `backend` must come from the same `StateManager.read()` snapshot, never a raw `state.json` parse plus a recovered backend read. BREAKS: scoped strict/final gates false-green when timeout/drift failures are misfiled as out-of-scope; orphan tmp recovery can otherwise run the strict gate in a stale repo while backend routing already follows the promoted state. ENFORCE: synthetic sentinel path in `anatomy-park-scoped-final-gate.test.js` and orphan-tmp working-dir recovery in `finalize-gate.test.js`.
- `src/services/backend-spawn.ts` — INVARIANT: every state-file-backed backend lookup (`resolveBackendFromStateFile()`, `loadBackendFromSession()`, and any caller that routes workers/remediators from `state.json`) must read through `StateManager.read()`, not raw `JSON.parse`. BREAKS: interrupted `state.backend` flips can strand the newer value in `state.json.tmp.<pid>` and silently route Morty/final-gate remediation through stale `claude` instead of `codex`. ENFORCE: orphan-tmp backend recovery in `backend-spawn.test.js` and the codex routing regression in `spawn-morty.test.js`.
- `src/bin/log-commit.ts` — INVARIANT: commit attribution must resolve the session for `process.cwd()` from `current_sessions.json` first, then fall back only to active sessions whose `state.json.working_dir` matches the same cwd; never pick the first active session globally. BREAKS: concurrent or cross-repo loops can stamp a commit onto the wrong session, and standup will confidently attribute that commit to the wrong task because the hook already set `event.session`. ENFORCE: multi-session cwd-mapping regression in `log-commit.test.js`.
- `src/bin/log-watcher.ts` — INVARIANT: session-liveness probes must read `state.json` through `StateManager.read()`, not raw `JSON.parse`, so dead-pid and orphan-tmp recovery can terminate the pane instead of tailing a dead session forever. BREAKS: the monitor log feed hangs on stale `active: true` state after a crash or interrupted write. ENFORCE: dead-pid termination regression in `log-watcher.test.js`.
- `src/bin/metrics.ts` — INVARIANT: user-visible/report date ranges are inclusive of the last represented day; if collection uses an internal upper bound, normalize it before writing `MetricsReport.until` or panel/no-data copy; CLI `--since` parsing must round-trip exact `YYYY-MM-DD` calendar components before scanning files, or impossible dates like `2026-02-30` normalize to March 2 and silently misreport the range; `--json` must always emit a `MetricsReport` object, even when there are zero rows. BREAKS: `--days 0` and default reports can claim tomorrow is in range even though no tomorrow rows were scanned; impossible `--since` dates silently shift the scan window to a different day; empty-data JSON callers crash on prose output instead of a parseable empty report. ENFORCE: CLI `--days 0`, impossible-calendar-date, and empty-report JSON regressions in `metrics.test.js`.
- `src/bin/morty-watcher.ts` — INVARIANT: session-liveness probes must read `state.json` through `StateManager.read()`, not raw `JSON.parse`, so dead-pid and orphan-tmp recovery can terminate the pane instead of tailing a dead session forever. BREAKS: the worker feed hangs on stale `active: true` state after a crash or interrupted write. ENFORCE: dead-pid termination regression in `morty-watcher.test.js`.
- `src/services/metrics-utils.ts` — INVARIANT: metrics report windows are closed local-day ranges on both collection legs: session JSONL scanning and git LOC scanning must both honor the same inclusive `since`/`until` day strings after bucketizing timestamps; git LOC must use the same canonical project slug as session-token storage, never basename/fuzzy suffix matching; metrics session-cache entries must carry the timezone that produced their local-day buckets, or unchanged JSONL files can keep reporting stale day splits after a timezone change. BREAKS: future-dated git commits can leak into reports that claim they end today, inflating rows and totals past `MetricsReport.until`; basename/suffix joins can stamp one repo's commit totals onto a different project's token row when names overlap; timezone-blind cache hits can pin a session file to the wrong report day until that file changes on disk. ENFORCE: future-dated git regressions in `metrics.test.js` for both `scanGitRepos()` and the CLI `--json` path, plus exact-slug vs fuzzy-match regressions in `buildReport` and the timezone-mismatched cache regression in `scanSessionFiles()`.
- `src/services/circuit-breaker.ts` — INVARIANT: circuit-breaker staleness checks must compare `last_progress_iteration` against the crash-recovered `state.json` snapshot from `StateManager.read()`, never a raw `JSON.parse`; orphan `state.json.tmp.<pid>` promotion can otherwise look like impossible CB drift and reset OPEN/HALF_OPEN back to fresh `CLOSED`. BREAKS: resumed mux runs can reopen a stalled loop and burn more worker iterations after a crash-interrupted state write. ENFORCE: orphan-tmp staleness regression in `circuit-breaker.test.js`.
- `src/services/pr-factory.ts` — INVARIANT: PR/session readers that derive repo path or prompt text from `state.json` must go through `StateManager.read()` crash recovery, not raw `JSON.parse`; otherwise orphan tmp promotion can already have fixed the session everywhere else while `gh pr create` still targets a stale `working_dir` and stale prompt. BREAKS: PR creation can run in the wrong repo and post the wrong session context after an interrupted state write. ENFORCE: orphan-tmp recovery regression in `pr-factory.test.js`.
- `src/bin/standup.ts` — INVARIANT: `range.until` is an exclusive upper bound for both activity-file selection and git commit collection and must never be printed directly; CLI `--since` parsing must round-trip exact `YYYY-MM-DD` calendar components before building the activity/git range, or impossible dates like `2026-02-30` normalize to March 2 and silently point the report at the wrong workday; after filename prefiltering, parsed activity events must still be re-checked against `[since, until)` by `event.ts` so legacy or externally mispartitioned JSONL lines cannot bleed into the wrong report; session project tags must read `sessions/<id>/state.json` via `StateManager.read()` so orphan `state.json.tmp.<pid>` recovery promotes the live `working_dir` before report formatting. BREAKS: standup headers and empty-range messages drift one day into the future while the reader still stops before that file; unbounded git-log collection leaks future-dated commits past the declared report end; impossible `--since` dates silently shift the report window; old/wrongly partitioned activity lines can show up in the wrong day even when the filename looks in-range; stale on-disk `working_dir` basenames can mislabel recovered sessions under the wrong project. ENFORCE: CLI `--days 0`, future-dated git commit, impossible-calendar-date, mispartitioned-activity, and crash-recovered-project regressions in `standup.test.js`.
- `src/bin/setup.ts` — INVARIANT: resume preflight checks that combine stored state with CLI flags must read through `StateManager.read()` before evaluating incompatible combinations like `backend === 'codex' && teams_mode === true`; raw parses can miss higher-iteration `state.json.tmp.<pid>` promotion and bless a forbidden resume state. BREAKS: interrupted writes can let `--resume` reactivate unsupported codex+teams sessions and push the manager pipeline into an invalid mode. ENFORCE: recovered-tmp conflict regression in `setup.test.js`.
- `src/bin/mux-runner.ts` — INVARIANT: every control-flow read of `state.json` in mux-runner and `runIteration()` must go through `StateManager.read()`; only the explicit crash-path `forceWrite` fallback may raw-parse the file; and the startup ownership flip (`active: false` → `true`) is valid only for fresh `tmux_mode` sessions created by `setup.ts`, never for generic inactive/recovered state. BREAKS: higher-iteration orphan tmp payloads with `active: false` or newer iteration metadata get ignored or reanimated, so the runner can keep executing stale work, validate dead command templates, or wait past a cancelled/inactive session. ENFORCE: orphan-tmp inactive regressions in `iteration-outcome.test.js` and `mux-runner.test.js`.
- `src/bin/raw-morty.ts` — INVARIANT: session-liveness probes must read `state.json` through `StateManager.read()`, not raw `JSON.parse`, so dead-pid and orphan-tmp recovery can terminate the pane instead of tailing a dead session forever. BREAKS: the raw feed hangs on stale `active: true` state after a crash or interrupted write. ENFORCE: dead-pid termination regression in `raw-morty.test.js`.
- `src/services/pickle-utils.ts` — INVARIANT: data-root resolution must honor canonical `PICKLE_DATA_ROOT` first and legacy `PICKLE_DATA_DIR` second, with both feeding every `getDataRoot()` caller consistently; session discovery must treat `current_sessions.json` as a cache and fall back to `sessions/*/state.json` by `working_dir`, because setup can leave a live session behind when the map write fails or goes stale; lookup reads must go through `StateManager.read()` crash recovery, not raw `JSON.parse`, so dead-PID `active: true` sessions are demoted before cache/scanner ranking; `pruneOldSessions()` must also read via `StateManager.read()` before honoring `active`, or dead-PID stale sessions pin old directories forever and setup never reclaims them; when the cache points at an existing session dir, explicit `state.json` evidence (`active: false`, unreadable/missing `state.json`, dead owning `pid`, or mismatched `working_dir`) must demote that mapped session below a live same-cwd fallback instead of shadowing it; a mapped session with missing/empty `working_dir` has not proved cwd ownership and may only survive as a compatibility fallback after the same-cwd scan, never outrank a scanned live session. ENFORCE: `getDataRoot: PICKLE_DATA_ROOT is the canonical override` in `pickle-utils.test.js`, the `PICKLE_DATA_ROOT` CLI path in `standup.test.js`, missing-map fallback regressions in `cancel.test.js`, `get-session.test.js`, `retry-ticket.test.js`, `status.test.js`, and `worker-setup.test.js`, plus the stale-mapped-inactive-session, dead-pid-session, unreadable-mapped-state, missing-working-dir-shadow, setup-prune regressions in `get-session.test.js`, `status.test.js`, `cancel.test.js`, `worker-setup.test.js`, and `setup.test.js`, and orphan-tmp jar recovery coverage in `jar-utils.test.js`.

ENFORCE: missing-map fallback coverage in `resolve-state.test.js` and `config-protection.test.js`, plus the stale-mapped-inactive-session, dead-pid-session, stale-env-state, and stop-hook regressions in `resolve-state.test.js`, `config-protection.test.js`, and `stop-hook.test.js`.
- `src/hooks/handlers/stop-hook.ts` — INVARIANT: after `resolveStateFile()` selects a session state file, stop-hook must read it through `StateManager.read()` rather than raw `JSON.parse`, so dead-PID crash recovery demotes stale `active: true` sessions before inactive/tmux/completion gating runs.
