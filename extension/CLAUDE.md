# Extension Subsystem

Compiled TS → JS lives in `extension/services/`, `extension/bin/`, `extension/hooks/`, `extension/types/`. Source in `extension/src/`. Tests in `extension/tests/` via `node --test`. Rebuild with `npx tsc` from `extension/`.

## Trap Doors

- `src/bin/council-publish.ts` — every `execFileSync('gh', …)` call must pass a `timeout` option; no-timeout calls block indefinitely on network hang; four prior "silent-failure hardening" passes missed this class because the existing tests mocked `gh` to respond instantly. New `gh` call sites must thread through `PublishOptions.ghTimeoutMs` or a per-call default, AND the test suite must cover the hang path with `__hang__` / `auth: 'hang'` / `hangOnCall` sentinels.
- `src/services/scope-resolver.ts` — every `spawnSync('rg' | 'grep', …)` call in `findImporters` must pass a `timeout` option (default `FIND_IMPORTERS_TIMEOUT_MS = 30_000`). A wedged ripgrep/grep (FIFO under repoRoot, stuck FUSE mount, catastrophic regex backtracking) would otherwise stall scope resolution indefinitely with no log signal. New tool-invocation sites in this file must thread `timeoutMs` through `computeOneHop({ findImportersTimeoutMs })` AND the test suite must cover the hang path with a fake tool script on `PATH` (see `scope-one-hop-hang-guard.test.js`).
- `src/bin/plumbus-frame-analyzer.ts` — every `spawnSync('bun', …)` call in `parseDotViaBun` must pass a `timeout` option (default `BUN_TIMEOUT_MS = 30_000`). A wedged bun (stuck registry import, infinite loop in `dump-graph.ts`, catastrophic regex on attractor graph traversal, stuck FUSE mount on the `.dot` target) would otherwise stall the plumbus generative-audit pipeline indefinitely — same silent-hang class as `council-publish` and `scope-resolver`. The existing `plumbus-frame-analyzer-bun.test.js` only covers exit-0 / exit-non-zero / missing-PATH paths with fake bun shims that respond instantly; new tool-invocation sites must keep the hang path covered by `plumbus-frame-analyzer-hang-guard.test.js` (hanging bun shim on `PATH`, wall-clock bound < `BUN_TIMEOUT_MS` + slack).
- `src/services/pickle-utils.ts` (`displayMacNotification`) — every macOS `osascript` shell-out from a runner exit path must route through `displayMacNotification` (or otherwise pass a `timeout` option, default `NOTIFICATION_TIMEOUT_MS = 5_000`). A wedged Notification Center / AppleEvent daemon (post-sleep-wake recovery, stuck TCC prompt, saturated `NSDistributedNotificationCenter`, display-notification queued at the lockscreen) will otherwise block `process.exit` on the runner with no signal and no log — same silent-hang class as `council-publish` gh, `scope-resolver` rg/grep, and `plumbus-frame-analyzer` bun, but fatal at shutdown (cancel markers never cleaned up, tmux pane prompt never returns). Five prior notification-touching commits (`8db2771`, `2f19356`, `f9f37ef`, `2da7fe5`, `8cc31a6`) missed this class because tests ran on Linux CI where `process.platform === 'darwin'` short-circuits the codepath entirely. New macOS-notification sites MUST call `displayMacNotification` rather than invoking `spawnSync`/`execFileSync` with `'osascript'` directly, AND the hang path must stay covered by `notification-hang-guard.test.js` (hanging shim via `spawnSyncFn` seam, wall-clock bound < `NOTIFICATION_TIMEOUT_MS` + slack).
- `src/bin/council-publish.ts` (directive/stack parity) — every non-trunk branch in `council-stack.json` MUST have a matching entry in `directive.branches`. A missing entry is NOT a silent "empty findings" case: it means the council fan-out dropped a shard (timeout, write error, skipped category) and the publisher must classify that branch as `outcome: 'failed'` with `error: 'council-directive.json has no entry for this branch — directive/stack mismatch'`, write NO body file, and stamp NO `.published` marker. v1.50.0 (900a43a) removed the prior heuristic "directive had content but produced zero per-branch findings" warning without adding a strict check; a directive with `branches: []` or a missing branch entry previously posted misleading "No findings for this branch at session close." comments to every affected PR AND stamped the re-run marker, permanently hiding the drift. Same silent-failure class as the spawnSync-no-timeout trap doors above. `validateDirective` in `council-schema.ts` enforces STRUCTURAL schema only — not set parity against `council-stack.json`; the parity check lives in the publisher's per-branch loop. New directive-consuming sites MUST surface directive/stack mismatch as `outcome: 'failed'` rather than defaulting to empty findings, AND the regression path must stay covered by the `stack branch absent from directive` test in `council-publish.test.js` (verifies no body file, no marker, only one `gh pr comment` call).
