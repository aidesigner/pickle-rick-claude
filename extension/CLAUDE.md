### Worker-convergence error contract

Worker-mode iteration timeouts enter the runner as `{ completion: 'error', timedOut: true, stallReason: 'wall_clock' }` for the 4-hour wall-clock guard and `{ completion: 'error', timedOut: true, stallReason: 'output_stall' }` for the 30-minute output-stall guard. `handleIterationOutcome(...)` routes those error outcomes through the same worker subprocess branch used for other worker-mode iteration errors.

In worker mode, consecutive subprocess failures count against `Defaults.WORKER_CONSECUTIVE_ERROR_CAP` (default `3`). `handleWorkerSubprocessError(...)` increments `consecutive_subprocess_errors`, returns `'continue'` while the count remains below `Defaults.WORKER_CONSECUTIVE_ERROR_CAP`, and returns `'error'` once the cap is reached. The counter resets only after a successful iteration.

The broader PRD contract for this branch also expects the surrounding subprocess-error plumbing to populate both `state.last_error` and `state.last_subprocess_error`, and to emit a `subprocess_error` activity event before the worker-mode cap logic is evaluated. Keep that dependency explicit when touching the worker error path so the manager-mode assumption from session `2026-05-10-6ed7182b` does not return.
