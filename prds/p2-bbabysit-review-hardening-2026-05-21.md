# R-CCR — B-BABYSIT-FIX review hardening

## Context

On 2026-05-21 a reliability bundle landed in three commits (`bf89a1a3`, `5501d4ed`,
`be269e03`) closing findings #46/#53/#54/#58-#64. A three-agent read-only review of
that diff (`9e0ae1d0..HEAD`) surfaced robustness gaps, one correctness bug, weak
exit-code semantics, and test-coverage holes. This PRD turns those review findings
into atomic, machine-checkable tickets.

All changes are in `extension/src/` (TypeScript; compiled mirrors in `extension/bin/`
are regenerated by `npx tsc`). Tests are `extension/tests/*.test.js` run via `node --test`.

## Tickets

### R-CCR-1 — `frontmatterCompletionCommitReachable` must fall back when the per-ticket workingDir is unusable

`extension/src/bin/mux-runner.ts` `frontmatterCompletionCommitReachable()` runs
`git -C <workingDir> merge-base --is-ancestor <sha> HEAD`. When `workingDir`
(derived from `ticket.working_dir`) does not exist or is not a git repo, git exits
non-zero and the stamped `completion_commit` is silently treated as unreachable —
so the phantom-Done watcher still reverts a genuinely-complete ticket. The doc
comment claims to close exactly this "stale workingDir" mode; it does not.

**Acceptance criteria**
- AC-CCR-1-1: When the per-ticket workingDir fails the reachability probe, the
  function retries against the session-level working dir (and/or `resolveRepoRoot`)
  before returning `false`.
- AC-CCR-1-2: A regression test constructs a ticket whose `working_dir` points at a
  non-existent path while the real repo (with the HEAD-reachable commit) is the
  session working dir; `correctPhantomDoneTickets` must NOT revert it.

### R-CCR-2 — `resolveRepoRoot` must resolve relative working_dirs against a stable base

`resolveRepoRoot(dir)` falls back to `path.resolve(dir)`, which resolves a relative
`working_dir` (e.g. `api/`) against `process.cwd()` of the mux-runner process — an
environment-dependent identity that can split or collide incorrectly.

**Acceptance criteria**
- AC-CCR-2-1: Relative `working_dir` values are resolved against a stable base (the
  session working dir / repo root), not raw `process.cwd()`.
- AC-CCR-2-2: A test with two tickets whose relative `working_dir`s denote the same
  repo resolves them to one root (no false multi-repo).

### R-CCR-3 — handoff exit_reason must only be honored on a clean (exit 0) phase exit

`extension/src/bin/pipeline-runner.ts` `runPhaseIteration` calls
`readHandoffExitReason` unconditionally. A genuinely failed phase (non-zero exit)
that left a stale `manager_handoff_pending`/`closer_handoff_terminal` in state would
be reclassified as a clean handoff, masking the failure.

**Acceptance criteria**
- AC-CCR-3-1: The handoff break in `runPhaseIteration` only fires when the phase
  runner exited with code 0.
- AC-CCR-3-2: A test with a non-zero phase exit plus a stale handoff exit_reason in
  state does NOT take the clean-handoff break path.

### R-CCR-4 — a clean handoff stop must not exit with the Failure code

When `manager_handoff_pending`/`closer_handoff_terminal` halts the pipeline,
`finalizePipeline` currently leaves `pipelineFailed=true` and the process exits with
`PipelineRunnerExitCode.Failure`. A documented clean handoff exiting as a failure is
incoherent.

**Acceptance criteria**
- AC-CCR-4-1: A pipeline that stops on a handoff exit_reason exits with `Success`
  (or a dedicated non-failure handoff code), not `Failure`.
- AC-CCR-4-2: `extension/tests/pipeline-runner-phase-no-progress.test.js` R-PRH test
  is updated to assert the corrected exit code.
- AC-CCR-4-3: `pipeline-status.json` for a handoff stop does not report `failed`.

### R-CCR-5 — decide and implement closer-release behavior for `closer_handoff_terminal`

Because handoff exits leave `pipelineFailed=true`, `buildCloserReleasePlan` /
`executeCloserReleasePlan` never run. For `closer_handoff_terminal` — whose purpose
is closer work — silently skipping the closer-release plan is likely wrong.

**Acceptance criteria**
- AC-CCR-5-1: The intended closer-release behavior for each handoff exit_reason is
  documented in a code comment in `finalizePipeline`.
- AC-CCR-5-2: If closer-release should run for `closer_handoff_terminal`, it does;
  a test asserts the chosen behavior.

### R-CCR-6 — `repairZeroWorkerTimeout` must not return a NaN `value`

For a non-repaired NaN input, `repairZeroWorkerTimeout` returns
`{ repaired: false, value: NaN }`. A caller trusting `.value` would propagate NaN.

**Acceptance criteria**
- AC-CCR-6-1: For a non-repaired result, `value` is never NaN — return the original
  numeric value if finite, else a documented sentinel.
- AC-CCR-6-2: A test asserts the non-repaired contract for `0`-absent, negative, and
  NaN inputs.

### R-CCR-7 — R-CCGR guard needs a negative-path test

`guardCompletionCommitBeforeDone`'s backoff re-read is only tested on the success
path. There is no test proving the guard still returns `ok:false` when the re-read
also finds no commit — i.e. that the backoff does not mask a genuinely commitless
ticket.

**Acceptance criteria**
- AC-CCR-7-1: A test (`rereadBackoffMs: 0`) asserts a ticket with no commit before
  AND after the re-read still yields `ok:false`, `source:'absent'`.

### R-CCR-8 — R-PDWR phantom-Done watcher test gaps

No test covers (a) a `completion_commit_inferred` frontmatter field as the second
checked field, nor (b) the discriminating case where a stamped SHA is NOT
HEAD-reachable and the revert therefore still proceeds.

**Acceptance criteria**
- AC-CCR-8-1: A test covers a ticket whose only evidence is `completion_commit_inferred`
  resolving to a HEAD-reachable commit — not reverted.
- AC-CCR-8-2: A test covers a ticket whose `completion_commit` SHA exists as an object
  but is NOT reachable from HEAD — the watcher still reverts it.

### R-CCR-9 — small test-coverage gaps

**Acceptance criteria**
- AC-CCR-9-1: `guardRereadBackoffMs` env handling is tested — `PICKLE_GUARD_REREAD_BACKOFF_MS`
  honored, the 5000ms ceiling clamped, NaN/negative rejected to the default.
- AC-CCR-9-2: `detectMultiRepo` has a test for the relative-`working_dir` fallback and
  a mixed git-repo + non-repo-dir session.
- AC-CCR-9-3: `check-readiness` `extractContractReferences` has a test pinning the
  intentional behavior for `@`-scoped package paths and `./`-prefixed paths.

### R-CCR-10 — handoff + incomplete-tickets interaction regression test

The R-PPPA and R-PRH fixes are tested in isolation. No test asserts that a clean
(exit 0) pickle exit carrying BOTH a `manager_handoff_pending` exit_reason AND
unresolved tickets preserves the handoff reason (not `phase_incomplete_tickets`).

**Acceptance criteria**
- AC-CCR-10-1: A test in `pipeline-runner-phase-no-progress.test.js` combines both
  conditions and asserts `exit_reason` stays `manager_handoff_pending`.

### R-CCR-11 — memoize `resolveRepoRoot` to cut redundant git spawns

`detectMultiRepo` (and the phantom-Done watcher path) spawn one `git` subprocess per
ticket. Memoize per call so repeated working_dirs resolve once.

**Acceptance criteria**
- AC-CCR-11-1: Within a single `detectMultiRepo` call, each distinct directory is
  resolved at most once (verified by a spy/counter test or by structural review).

### R-CCR-12 — document PATH_RE's intentional `@`-scoped-path exclusion

The R-RHFP `PATH_RE` negative lookbehind `(?<![\w./@-])` deliberately excludes
`@`-prefixed scoped-package paths. Add a one-line comment so a future reader does not
"fix" it as a bug.

**Acceptance criteria**
- AC-CCR-12-1: `check-readiness.ts` near `PATH_RE` has a comment stating
  `@`-scoped-package paths are deliberately excluded (not in-repo refs).

### R-CCR-13 — close R-RHFP sub-fix (c): inline code-snippet false positives

Finding #64 R-RHFP sub-fix (c) was deferred: `check-readiness` resolves inline
code-snippet tokens (`t.skip()`, `t.mock.timers.enable()`, `inputs.run_count`) as if
they were file/symbol references, producing `performance`-timeout false positives.
Investigate and implement a safe way to skip obvious snippet tokens WITHOUT
suppressing legitimate backticked contract references.

**Acceptance criteria**
- AC-CCR-13-1: A ticket body quoting `t.skip()` / `t.mock.timers.enable()` /
  `inputs.run_count` as inline code produces zero `performance` or `contract`
  findings for those tokens.
- AC-CCR-13-2: A ticket body citing a genuine in-repo backticked symbol still
  produces a finding when that symbol does not resolve (no over-suppression).
- AC-CCR-13-3: A fixture test in `extension/tests/` proves both AC-CCR-13-1 and -2.
